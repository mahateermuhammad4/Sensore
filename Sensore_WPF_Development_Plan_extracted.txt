1. Project Overview

Sensore is a WPF desktop application developed for Graphene Trace, a MedTech startup based in Chelmsford, UK. The system processes real-time pressure sensor data from a 32×32 e-textile mat used to prevent pressure ulcers in wheelchair users and immobile patients. This development plan covers every aspect of building the application — from environment setup to final deployment — adapted specifically for MySQL Workbench 8.0 CE as the database engine.

1.1 What the Application Must Do
Import and parse time-stamped CSV files containing 32×32 pressure frames (1,024 values per frame)
Analyse each frame to compute Peak Pressure Index (PPI) and Contact Area percentage
Detect dangerous pressure conditions and raise real-time alerts after 3 consecutive high-PPI frames
Render a live colour-coded 32×32 heatmap on the patient dashboard
Display time-series charts for PPI and Contact Area over 1h, 6h, 24h, and 7d windows
Support three distinct user roles: Patient, Clinician, and Administrator
Allow threaded comments between patients and clinicians linked to specific pressure frames
Generate PDF summary reports for a selected patient and date range
Store all data securely in MySQL with BCrypt-hashed passwords

1.2 Key Technology Decisions
Why MySQL Instead of SQLite?
The original requirements document specifies SQLite. However, since MySQL Workbench 8.0 CE is
already installed on your machine, this plan adapts the architecture to use MySQL 8.0.

Advantages of MySQL for this project:
  • MySQL Workbench gives you a full visual schema designer, query editor, and data browser
  • MySQL handles larger data volumes more efficiently as the SensorFrame table grows
  • The EF Core migration workflow is identical — only the connection string and NuGet package differ
  • MySQL supports indexed REAL columns for fast PPI/ContactArea time-series queries

The change requires: Pomelo.EntityFrameworkCore.MySql (NuGet) instead of EF Core SQLite.
All LINQ queries, migrations, and model code remain exactly the same.

5.3 ICommand Implementation (RelayCommand)
Create one reusable RelayCommand class that wraps any Action. All ViewModel commands use this:
RelayCommand.cs — Place in ViewModels/ folder
public class RelayCommand : ICommand {
    private readonly Action<object> _execute;
    private readonly Func<object, bool> _canExecute;
    public RelayCommand(Action<object> execute, Func<object,bool> canExecute = null) {
        _execute = execute; _canExecute = canExecute;
    }
    public bool CanExecute(object p) => _canExecute == null || _canExecute(p);
    public void Execute(object p) => _execute(p);
    public event EventHandler CanExecuteChanged {
        add => CommandManager.RequerySuggested += value;
        remove => CommandManager.RequerySuggested -= value;
    }
}

6. Service Layer — Detailed Implementation

6.1 AuthService
Handles all authentication. Never stores or logs plaintext passwords.
Method
Logic
Authenticate(email, password)
1. Query Users WHERE Email = email. 2. If null or !IsActive → return null. 3. Call BCrypt.Verify(password, user.PasswordHash). 4. If false → return null. 5. Return the full User entity with Role set.
CreateUser(email, name, role)
1. Generate hash: BCrypt.HashPassword(password, workFactor: 12). 2. Create User entity. 3. SaveChangesAsync. Admin-only operation.

6.2 CsvImportService
Reads CSV files from disk, parses 32-row frames, runs analysis, then bulk-saves to MySQL.
Read file line-by-line; collect every 32 lines as one frame
Split each line by comma; parse each of the 32 tokens as integers (1–255)
If any token fails to parse: skip that row, log error, do not discard whole frame
On completing 32 valid rows: call PressureAnalysisService.CalculatePPI and CalculateContactArea
Call AlertService.EvaluateAsync with UserId and PPI
After ALL frames processed: single SaveChangesAsync — atomic batch commit to MySQL
Return ParseResult { int FramesSaved, List<string> Errors }

6.3 PressureAnalysisService — Algorithm Details
6.3.1 PPI Calculation (BFS Flood Fill)
Parse the FrameData string into a 32×32 int[,] array
Initialise bool[32,32] visited = all false
Define lower threshold: const int THRESHOLD = 50
For each pixel (r, c) in the 32×32 grid:
  If visited[r,c] is true → skip
  If frame[r,c] <= THRESHOLD → mark visited, skip
  Start BFS: enqueue (r,c). regionCount=0, regionMax=0
  While queue not empty: dequeue pixel, check 4 neighbours (up/down/left/right)
  If neighbour is in bounds, not visited, and > THRESHOLD: enqueue, mark visited, track count and max
  After BFS: if regionCount >= 10 → candidate = max(candidate, regionMax)
Return the highest candidate value found. Return 0 if no qualifying region exists.

6.3.2 Contact Area Calculation
Linear O(n) scan: iterate all 1,024 pixels. Count pixels where value > THRESHOLD. Return (count / 1024.0) * 100 as a double percentage.

6.4 AlertService
Maintain Dictionary<int, int> consecutiveCounts keyed by UserId
On EvaluateAsync(userId, ppi): if ppi > 200 → increment counter; else reset to 0
When counter reaches 3: create Alert entity, set IsFlagged = true on the frame, raise AlertTriggered event
AlertTriggered event: PatientDashboardViewModel subscribes and sets IsAlertVisible = true
AcknowledgeAlertCommand: sets IsAcknowledged = true in DB and hides the banner

6.5 ReportService
Query SensorFrames for patientId between DateFrom and DateTo
Compute: average PPI, peak PPI, average ContactArea, peak ContactArea for the period
Compute day-on-day comparison: average PPI today vs yesterday
Compute 'this hour vs same hour yesterday' metric
Use QuestPDF to generate a formatted PDF with patient name, date range, metrics table, and trend summary
Save generated PDF to application data folder; return file path to ViewModel for display

7.3 Patient Dashboard — Heatmap Implementation
The 32×32 heatmap uses a WPF UniformGrid with 1,024 Rectangle elements:
ObservableCollection<HeatmapCell> where HeatmapCell has a Color property (SolidColorBrush)

Color mapping function: value 1 → Blue (#0000FF), value 128 → Yellow (#FFFF00), value 255 → Red (#FF0000)
Use linear interpolation through the colour spectrum for intermediate values
When new CSV is imported, update all 1,024 cells in one batch — do NOT update individually to avoid flicker
UniformGrid Rows=32 Columns=32, each cell approximately 10×10 px for a 320×320 total canvas

10. Security Implementation

10.1 Password Security (BCrypt)
Use BCrypt.Net-Next package — never SHA256, MD5, or plain text
Work factor 12 recommended — balances security vs login speed on modern hardware
On account creation: BCrypt.HashPassword(password, workFactor: 12) → store the 72-char hash string
On login: BCrypt.Verify(submittedPassword, storedHash) — returns bool
Never log, display, or pass the plaintext password anywhere in the codebase
MySQL Workbench confirmation: PasswordHash column should always show $2a$ or $2b$ prefix

10.2 Role-Based Access Control
Roles are enforced at the ViewModel and Service layer — never relying on UI alone
Patient queries: always filter WHERE UserId = loggedInUser.UserId
Clinician queries: always JOIN ClinicianPatients WHERE ClinicianId = loggedInUser.UserId
Admin: no access to clinical data — only User management tables
MainViewModel refuses to navigate to a dashboard if Role does not match

10.3 MySQL Connection Security
Use a dedicated MySQL user (sensore_user) with only SELECT, INSERT, UPDATE on sensore_db
Do NOT connect as root from the application
Store the connection string in an app config file — exclude this file from Git (.gitignore)
For production: consider encrypting the connection string using DPAPI
