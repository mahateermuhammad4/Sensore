1. Problem Definition and Software Requirements
1.1 Case Study Overview
Graphene Trace is a 
MedTech
 startup based in Chelmsford, UK, developing a product ca
lled 
Sensore
 — a continuous, automated 
pressure ulcer prevention system
. The system uses an e-textile mat embedded with pressure sensors that generate real-time, 32×32 
grid pressure distribution maps
. These maps 
are saved
 as time-stamped CSV files and are 
designed to be read by monitoring software that can detect dangerous pressure zones and alert caregivers before a pressure ulcer develops.
Pressure ulcers (also known as bedsores or decubitus ulcers) are a preventable but serious patient safety concern, particularly for wheelchair users and immobile patients. Each CSV file produced by the 
Sensore
 hardware stores one or more 32×32 pressure fra
mes, where each row corresponds to one row of sensors and each value represents the pressure at that sensor pixel. Values range from 
1
 (zero applied force) up to 255 (maximum saturation), scaling linearly. Files 
are named and 
organised
 by patient user ID a
nd date/time, providing a structured dataset that the software application must ingest, store, 
analyse
, and 
visualise
.
1.2 Current System Description
Before this software project, there is no dedicated tool to manage, process, or display the data generate
d by the 
Sensore
 mat. Raw CSV files accumulate in a folder structure on disk, 
organised
 loosely by user ID and date. There is no central dashboard, no alert mechanism, no user role separation, and no reporting capability. Clinical staff would need to open 
raw files manually and 
read through
 hundreds of rows of comma-separated numbers to detect any problematic pattern — a process that is simply not workable at scale. This project exists specifically to replace that manual process with an automated, role-base
d, data-driven application.
1.3 Software Requirements
Functional Requirements
FR1 — 
Database:
 Design and implement a structured SQLite database to store time-ordered 32×32 pressure map data per patient, along with computed metrics and associated metadata.
FR2 — User Roles: Support three distinct login types. Patients see only their own data. 
Clinicians can view all or selected assigned patients' data.
 Administrators create and manage all user accounts.
FR3 — Pressure Analysis and Alerts: 
Analyse
 each incom
ing pressure frame to identify high-pressure regions. Generate on-screen alerts for the patient and flag the event in the database for clinician review.
FR4a — Peak Pressure Index (PPI): Calculate the 
highest pressure
 value within any contiguous region of 
ten or more active pixels in a frame. Regions with fewer than ten pixels 
must be excluded
 from the calculation.
FR4b — Contact Area Percentage: Compute the proportion of all 1,024 pixels that exceed a defined lower threshold, expressing this as a percentag
e of total mat coverage.
FR5 — Time-Series Graphs: Display PPI and Contact Area as line graphs over selectable time windows — the last one hour, last six hours, last 24 hours, and last seven days.
FR6 — Report Generation: Produce summary reports for a chos
en patient and date range, including key metric averages, peak values, and comparisons (day-on-day and this hour versus the same hour yesterday).
FR7 — Comment and Feedback System: Allow patients to submit timestamped comments linked to specific pressure f
rames. Clinicians can view these comments and reply in a threaded format.
Non-Functional Requirements
NFR1 — Security: User passwords must be stored as salted hashes using 
BCrypt
. The application must never store or log plaintext credentials.
NFR2 — Perfo
rmance: The main dashboard must load and render 
heatmap
 data within three seconds for a standard 24-hour dataset.
NFR3 — Usability: The interface must be clear and understandable to non-technical users including patients and clinical staff.
NFR4 — Scalabil
ity: The database schema and service layer must support growth in the number of patients and data volume without architectural redesign.
NFR5 — Maintainability: All code must follow the MVVM pattern, 
be clearly commented
, and be free of redundant or unused
 code.
Nice-to-Have Requirements
NTH1 — Additional Metrics: Extract further pressure analysis values beyond PPI and Contact Area, such as 
centre
-of-pressure shift or left/right asymmetry index.
NTH2 — Enhanced 
Visualisation
: Improve the visual quality of 
heatmaps
 and charts using smooth 
colour
 gradients and animated transitions accessible to lay audiences.
1.4 Project Aims
The primary aim is to deliver a functional, secure, and clinically useful WPF desktop application that enables Graphene Trace to move beyond raw CSV files and operate with a proper data management and monitoring system. More specifically, the project aims 
to:
Give patients real-time visibility of their own pressure data and automatic alerts when prolonged high pressure 
is detected
.
Allow clinicians to efficiently monitor multiple patients, review flagged pressure events, and engage with patient comments thr
ough a threaded interface.
Enable administrators to create, edit, and deactivate user accounts from a dedicated management interface.
Generate structured, readable clinical reports that support informed care decisions and allow trend comparison over time.
Build a local SQLite-backed data architecture capable of ingesting and querying large volumes of sensor CSV data reliably and efficiently.

2.2.3 Design Rationale
Storing the raw 32×32 frame data as a comma-separated text string in the 
FrameData
 column keeps the schema stable regardless of future changes to sensor resolution. The computed metric columns (PPI, 
ContactArea
) are stored as indexed numeric fields so that
 time-series queries for charting can run efficiently against them without needing to re-parse any raw data at query time.
The self-referencing 
ParentCommentId
 foreign key in the Comments table enables a threaded reply structure without requiring a separat
e table. The 
IsFlagged
 column on 
SensorFrames
 acts as a fast 
boolean
 filter for the clinician's flagged events list, avoiding a join to the Alerts table in the most common query path.

3. Software Implementation Documentation
3.1 Application Architecture
The application is a WPF desktop program built on .NET 8, following the MVVM pattern. The solution 
is divided
 into four logical layers:
Data Layer: 
AppDbContext
 with EF Core (SQLite) manages all database reads and writes. No raw SQL 
is written
; all schema 
changes go through EF Core migrations.
Service Layer: Business logic lives entirely in service classes. These classes have no knowledge of WPF or any UI concept — they accept plain .NET types and return results that the 
ViewModel
 can consume.
ViewModel
 Layer: 
ViewModels
 expose 
ObservableCollection
<T> for list data and 
ICommand
 objects for actions. When a service returns data, the 
ViewModel
 updates its properties, and WPF's data binding engine propagates the changes to the View automatically.
Vi
ew Layer: XAML files define layout, styling, and data bindings only. The only code permitted in a code-behind file is the 
DataContext
 assignment in 
MainWindow
.
Dependencies 
are injected
 via constructors. 
App.xaml.cs
 creates all service instances at startup
 and passes them into 
MainViewModel
. Child 
ViewModels
 receive only the services they actually need, keeping coupling low.

3.4.2 Peak Pressure Index Algorithm
The PPI 
algorithm uses a breadth-first search (BFS) flood-fill on the 32×32 pixel array. A 
boolean

visited[
][] array is 
initialised
 to track which pixels have already been processed. For each unvisited pixel whose value exceeds the lower threshold (default: 50), a
 BFS queue is seeded with that pixel. The BFS explores all four-connected 
neighbours
 (up, down, left, 
right
) above the threshold, recording the count of pixels in the region and the maximum value found. After the BFS completes, if the region contains at le
ast ten pixels, its maximum value 
is registered
 as a candidate for PPI. Regions of fewer than ten pixels 
are discarded
. When all pixels 
have been visited
, the highest registered candidate value is returned as PPI. If no qualifying region 
exists
 the method 
returns zero.

3.4.3 Contact Area Calculation
Co
ntact Area 
is computed
 by iterating all 1,024 pixels and counting those whose value exceeds the lower threshold. The count 
is then divided by 1,024.0 and multiplied by 100 to produce a percentage
. This 
is a simple linear scan that
 runs in constant time rel
ative to the fixed grid size and requires no separate data structure.
3.4.4 Alert Detection
AlertService
 maintains an in-memory Dictionary<
int
, 
int
> keyed by 
UserId
 that tracks the current consecutive high-PPI frame count for each patient. Each time a new
 frame is imported, 
EvaluateAsync

is called
 with the patient's 
UserId
 and the frame's computed PPI. If PPI exceeds the 
alertThresholdHigh
 (default: 200), the counter increments; if it falls below the threshold, the counter resets to zero. When the counter 
reaches three, an Alert entity is created with

 =

,

SensorFrame
, and the

 event is raised. The

 = true.

5. Plan of Software Maintenanc
e
5.1 Maintenance Philosophy
The 
Sensore
 application sits in a safety-adjacent clinical context. Any defect that produces incorrect PPI values, silently fails to raise an alert, or prevents a clinician from accessing a patient's data could have real conse
quences for patient care. The maintenance plan therefore 
prioritises
 rapid defect response, transparent version control, and structured change management. The approach follows the framework defined in ISO/IEC 14764, which 
categorises
 maintenance activities
 as corrective, adaptive, perfective, or preventive.
5.2 Categories of Maintenance
Type
Description
Example Activities
Corrective
Fixing defects discovered after deployment.
Correcting a PPI calculation error; fixing a login navigation bug; resolving a crash when a CSV file uses Windows CRLF line endings.
Adaptive
Updating the system for environmental changes.
Upgrading to a newer .NET LTS version; updating EF Core; adapting the CSV parser if Graphene Trace releases new hardware with a different file format.
Perfective
Enhancing functionality based on user feedback.
Adding the nice-to-have additional pressure metrics (NTH1); improving 
heatmap

colour
 gradients (NTH2)
;
 adding an Excel export option for reports.
Preventive
Proactive work to prevent future failures.
Refactoring the CSV parser for whitespace resilience; adding a 
database connection retry policy
; introducing a startup health check that validates the schema version.
5.3 Version Control and Branching Strategy
All source code 
is kept
 in a private GitHub repository using 
Git
 Flow. This model provides clear separation between production-ready code and work in progress:
main
 — contains only production-ready, fully tested releases. All merges require a pull request 
with at least one peer review.
develop
 — the integration branch. All feature and fix branches merge here first and must pass automated tests before merging.
feature
/[name] — one branch per user story or enhancement, branched from develop.
hotfix
/[name] — c
ritical bug fixes branched from main; merged back into both main and develop on completion.
All tagged releases use semantic versioning (MAJOR.MINOR.PATCH). EF Core migration files are committed alongside the code change that requires them, so any develope
r can bring their local database up to date by running the EF CLI update command.
5.4 Bug Reporting and Resolution Process
A bug 
is reported
 via a GitHub Issue (development team) or a feedback form within the application (clinicians and patients).
The iss
ue 
is triaged within 24 hours and assigned a severity
: Critical, High, Medium, or Low.
Critical (data corruption or system unavailable) addressed within one working day. High within three working days. Medium within two weeks. Low in the next scheduled sprint.
A fix 
is developed on a hotfix or feature branch, peer-reviewed, and merged
.
A reg
ression test 
is added
 for the specific defect before the issue is closed, to prevent recurrence.
5.5 Dependency and Framework Updates
A monthly scheduled review will check all 
NuGet
 packages for known security vulnerabilities. The .NET SDK 
will be updated
 to the latest Long-Term Support release within three months of its official publication. Third-party packages (
OxyPlot
, BCrypt.Net-Next, EF Core, 
QuestPDF
) are pinned to specific version numbers and updated only after running the full test suite. Any upda
te that includes a breaking API change requires a corresponding code migration and a changelog entry.
5.6 Database Maintenance
All schema changes 
are made
 exclusively through EF Core migrations. No manual SQL edits against the database file 
are permitted
.
The SQLite database file 
is backed
 up daily by a scheduled copy to a designated backup folder. For future network deployment, automated remote backup would replace this.
SensorFrame
 records older than two years 
will be moved
 to a cold-storage archive tabl
e to maintain query performance as the dataset grows.
Database integrity is verified monthly using the SQLite PRAGMA 
integrity_check
 command as part of a maintenance script.
5.7 Security Maintenance
The 
BCrypt
 work factor 
is reviewed
 annually. If the fact
or 
is increased
 to counter hardware advances, users are prompted to re-authenticate at their next login, which rehashes their password with the new factor.
Dependency vulnerability scans 
are run
 on every commit via a GitHub Actions workflow. A critical vul
nerability in any dependency triggers an immediate adaptive maintenance task.
If the application 
is ever extended
 to a networked deployment, SSL/TLS certificate renewal must be automated and monitored as part of the deployment pipeline.
5.8 Documentation Maintenance
Every pull request must include updated inline comments where the change affects non-obvious logic. The GitHub Wiki contains user guides for each role and a developer setup guide; these 
are reviewed and updated after every sig
nificant UI or feature change
. XML documentation comments on all public service methods are kept current and would power a generated API reference document if the application 
is
 extended with a REST API layer.
5.9 Monitoring and Health Checks
On every app
lication startup, a health-check routine runs before the login screen 
is shown

gured

The maintenance plan described in this section 
is designed

Sensore
